Project document

A fintech-style payment backend built to explain, defend, and showcase.

I built a Spring Boot payment orchestration service that accepts payment requests, routes them to the right provider, protects against duplicates with idempotency, persists durable state in MySQL, and exposes Prometheus/Grafana observability.

Tech stack

  • Java 21
  • Spring Boot
  • Spring Data JPA + Redis
  • Spring Retry + Resilience4j
  • Micrometer / Prometheus / Grafana
  • MySQL
  • OpenAPI / Swagger

Core design idea

The user chooses a payment method like debit card, credit card, UPI, or net banking. The backend chooses the provider, retries transient failures, and falls back to a secondary provider if the primary one is down.

End-to-end request flow

1. Client submits payment

The request includes an idempotency key and payment method.

2. Controller validates input

Request validation ensures the API is safe before orchestration begins.

3. Idempotency service checks Redis + DB

It detects duplicate requests and reuses stored responses when needed.

4. Routing strategy picks provider

The first provider is selected based on the payment method.

5. Provider executor applies retry + timeout

Transient provider issues are retried automatically.

6. Failover handles provider outage

If the primary provider fails, the orchestration layer tries the fallback provider.

7. Final state is saved

The payment status, provider, and retry count are persisted in MySQL.

8. Metrics are recorded

Latency, retries, and outcomes are exposed through Micrometer and Prometheus.

How it scales

  • API layer is stateless, so replicas can scale horizontally.
  • Redis handles fast duplicate detection and in-flight locks.
  • MySQL remains the durable source of truth.
  • Circuit breakers prevent one bad provider from hurting the whole service.
  • Prometheus/Grafana let you observe traffic, retries, and latency.

Interview answers

  • Why idempotency? To prevent duplicate charges on retries.
  • Why Redis? For low-latency locks and cached responses.
  • Why MySQL? For durable payment state and auditability.
  • Why failover? To continue processing when a provider is unavailable.
  • Why Grafana? To see live payment throughput and retry patterns.

How to explain the impact

This project shows more than CRUD. It demonstrates the exact concerns that matter in fintech: correctness under retries, safe provider switching, concurrency handling, observability, and a clean path to horizontal scaling.
