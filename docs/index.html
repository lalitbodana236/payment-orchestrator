Payment orchestration built for reliability, scale, and interview depth.

This project simulates a production-style payment backend: it routes payments to the right provider, enforces idempotency, handles retries and failover, persists state in MySQL, uses Redis for hot-path deduplication, and exposes Prometheus/Grafana observability.

Why this project stands out

Provider routing Automatic failover Idempotency keys Redis locks MySQL persistence Metrics & tracing Swagger UI
  • Routes `DEBIT_CARD`, `CREDIT_CARD`, and `NET_BANKING` to the primary card provider.
  • Routes `UPI` to the UPI provider first, then automatically fails over if needed.
  • Prevents duplicate payment execution with idempotency locks and cached responses.
  • Exposes custom app metrics plus Prometheus and Grafana dashboards for visibility.

Interview pitch

“I built a Spring Boot payment orchestration service that safely handles payment retries, duplicate requests, provider failures, and concurrent traffic. The backend stays stateless so it can scale horizontally, while Redis and MySQL split hot-path deduplication from durable persistence.”

Architecture

Client → Controller → Orchestration Service → Routing Strategy → Provider Executor → Provider Connector → MySQL / Redis

What happens on a request

1. Validate

Checks input, correlation ID, and idempotency key.

2. Route

Picks the provider based on payment method.

3. Execute

Retries transient failures and handles timeouts.

4. Persist

Saves final payment status and caches the response.
