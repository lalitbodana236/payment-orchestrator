The problem

Payment systems often fail because of retries, provider downtime, duplicate requests, or inconsistent state. In fintech, a request must either finish correctly or fail safely without creating duplicate transactions.

The impact

This architecture reduces duplicate charges, preserves payment integrity during retries, and gives operators visibility into latency, failures, and retry storms.

How the solution works

Idempotency Routing Retry Failover Persistence Observability
  • The user selects a payment method such as debit card, credit card, UPI, or net banking.
  • The backend chooses the primary provider automatically and hides provider internals.
  • If the primary provider fails, the orchestration layer retries and then fails over to a backup provider.
  • Payment state is stored in MySQL; duplicate request control uses Redis.

Strong interview line

“I built a payment orchestration service that makes sure each payment is processed once, even if requests are retried, providers fail, or traffic spikes. The service stays stateless so it can scale horizontally.”

Architecture

Client → Controller → Orchestration Service → Routing Strategy → Provider Executor → Provider Connector → MySQL / Redis

Request flow

1. Validate

Checks request body and idempotency key.

2. Route

Selects the primary provider by payment method.

3. Execute

Retries transient failures and handles timeout.

4. Persist

Saves the final payment result and metrics.
